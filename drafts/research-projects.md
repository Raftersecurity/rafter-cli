# Security Research Projects Using Rafter CLI

Three original research studies examining the security posture of AI agent ecosystems. Each is designed to produce publishable, citable findings using Rafter CLI as the primary analysis instrument.

---

## 1. The State of Secrets in AI Agent Configurations

### Project Title
**Secrets in the System Prompt: A Large-Scale Audit of MCP and AI Agent Configuration Files on Public GitHub**

**Research Question:** How prevalent are hardcoded secrets in publicly committed AI agent configuration files, which platforms and secret types are most affected, and how does the rate compare to traditional application configuration files?

**Why This Matters:** GitGuardian's 2025 report found 24,008 secrets in MCP configuration files on public GitHub, but provided no breakdown by platform, secret type, or comparison baseline. AI agent configs are uniquely dangerous because they often contain API keys with broad permissions (LLM provider keys, tool-use credentials, database connection strings) and are frequently copy-pasted from tutorials or generated by setup wizards that don't warn about secrets. A rigorous, reproducible study with per-platform and per-pattern breakdowns would be the first of its kind and would give platform vendors specific data to act on.

**Methodology:**

- **Data collection:** Use the GitHub Search API and `gh` CLI to collect configuration files associated with the 8 major AI agent platforms. Target filenames and paths:
  - Claude Code: `.claude/settings.json`, `claude_desktop_config.json`, `.claude/CLAUDE.md`
  - Cursor: `.cursor/mcp.json`, `.cursor/rules/*.mdc`
  - Windsurf: `.windsurf/mcp.json`, `.windsurfrules`
  - Continue.dev: `.continue/config.json`, `.continue/config.yaml`
  - Codex: `codex.json`, `AGENTS.md`
  - Gemini: `GEMINI.md`
  - Aider: `.aider.conf.yml`, `.aider/conventions.md`
  - Generic MCP: `mcp.json`, `mcp-config.json`, `mcp-servers.json`

  Sampling strategy: Collect all public files matching these patterns (expected: 50,000-200,000 files based on GitGuardian's numbers). Deduplicate by content hash to remove forks. Final corpus target: 30,000+ unique configuration files.

- **Analysis tools:**
  - Primary: `rafter scan local --format json` on each file, using all 21 built-in secret patterns (AWS, GitHub, Google, Slack, Stripe, Twilio, npm, PyPI, database connection strings, private keys, JWTs, bearer tokens, generic API keys/secrets)
  - Secondary: Gitleaks (via Rafter's binary integration) for cross-validation with an independent pattern set
  - Tertiary: Manual review of a stratified random sample (n=500) to establish ground-truth precision and recall

- **Controls:**
  - **False positive mitigation:** Exclude files in directories named `test`, `example`, `fixture`, `mock`, `template`. Flag any finding where the secret value matches known placeholder patterns (e.g., `sk_live_xxxx`, `your-api-key-here`, strings of all zeros). Report both raw and filtered counts.
  - **Dual-engine validation:** A finding is "confirmed" only if both Rafter's regex engine and Gitleaks flag it, OR if it passes the manual review sample. Report single-engine-only findings separately.
  - **Baseline comparison:** Collect a size-matched control corpus of traditional config files (`.env`, `config.yaml`, `settings.json` NOT in AI agent directories) from the same time period. Run identical scans. This provides the "is AI config worse than regular config?" comparison.
  - **Statistical approach:** Chi-squared tests for platform-to-platform prevalence differences. 95% confidence intervals on all rates. Cohen's h for effect size on AI vs. traditional config comparison.

- **Timeline:** 3-4 weeks. Week 1: data collection and deduplication. Week 2: automated scanning of full corpus. Week 3: manual validation sample, statistical analysis, baseline comparison. Week 4: write-up and visualization.

**Expected Findings:** We expect to confirm GitGuardian's order-of-magnitude finding (tens of thousands of secrets) and add new dimensions: (a) a per-platform ranking showing which AI agent ecosystems have the worst secret hygiene, (b) a per-secret-type breakdown showing that LLM provider API keys (OpenAI, Anthropic, Google) dominate but that database credentials and cloud provider keys also appear at concerning rates, (c) evidence that AI agent configs leak secrets at 2-5x the rate of traditional application configs, likely because MCP setup workflows encourage pasting API keys directly into JSON. There is genuine uncertainty about which platform will rank worst -- Cursor and Claude Code have the largest install bases, but smaller platforms may have less mature tooling and worse hygiene.

**Publication Strategy:**
- Primary: Long-form technical blog post with interactive charts (platform breakdown, secret type heatmap, time series if commit dates are available)
- Secondary: Preprint on arXiv (cs.CR) for academic citation
- Amplification: Submit to Hacker News, post thread on X/Twitter, pitch to security journalists at The Record, Bleeping Computer, and Dark Reading
- Data release: Publish the scanning methodology, Rafter commands used, and aggregate statistics (never raw secrets or repo identifiers)

**Ethical Considerations:**
- **No secret values published.** All findings are reported as counts and rates, never raw credential strings.
- **Responsible disclosure:** Before publication, notify the platform vendors (Anthropic, Cursor, Codeium, Continue.dev) with aggregate findings so they can add warnings to their setup flows. Allow 30 days for response.
- **No repo shaming.** Individual repositories are never identified. Aggregate data only.
- **GitHub ToS compliance:** All data is from public repositories. Automated collection respects rate limits and robots.txt norms.
- **Secret validity:** We do NOT test whether discovered secrets are still active. No authentication attempts are made.

## 2. Do AI Coding Assistants Leak Secrets? A Comparative Audit of AI-Generated vs. Human-Written Code

### Project Title
**Copy, Paste, Expose: Measuring Secret Leakage Rates in AI-Generated Code Contributions**

**Research Question:** Do commits attributed to AI coding assistants contain hardcoded secrets and credentials at a higher rate than human-written commits in the same repositories, and which secret types are disproportionately introduced by AI?

**Why This Matters:** AI coding assistants (Copilot, Claude, Cursor, Codex) are now responsible for a significant and growing share of new code. Anecdotal evidence suggests these tools sometimes generate code containing placeholder credentials, example API keys, or secrets copied from training data. However, no large-scale empirical study has measured whether AI-attributed commits actually leak secrets at a different rate than human commits. This study would be the first controlled comparison, providing evidence that either vindicates or indicts current AI coding practices.

**Methodology:**

- **Data collection:** Identify AI-attributed commits using multiple signals:
  - `Co-Authored-By` trailers mentioning Copilot, Claude, Cursor, Codex, or known AI bot accounts
  - Commit messages containing "generated by", "AI-assisted", "copilot", "claude", or similar attribution strings
  - GitHub's Copilot attribution metadata (where available via API)

  Target: 100,000 AI-attributed commits and 100,000 human-only commits from the same repositories (matched control). Restrict to repositories with at least 100 commits and at least 5 contributors to ensure established projects. Languages: Python, JavaScript/TypeScript, Go, Java, Rust (top 5 by AI tool adoption).

  For each commit, extract the diff (added lines only -- we care about what was introduced, not what was present before).

- **Analysis tools:**
  - Primary: `rafter scan local --format json` on each diff, targeting all 21 secret patterns. The diff-only approach is critical: scanning added lines isolates what the commit introduced.
  - Secondary: Gitleaks on the same diffs for cross-validation.
  - Custom: A lightweight classifier to categorize findings as "likely real secret" vs. "example/placeholder" vs. "test fixture" based on entropy analysis and context (surrounding code, file path, variable names).

- **Controls:**
  - **Matched-pair design:** For each AI-attributed commit, select a human commit from the same repository, same language, and similar diff size (within 2x lines changed). This controls for project-level secret hygiene, language norms, and repository maturity.
  - **False positive filtering:** Apply the placeholder classifier to all findings. Report results at three thresholds: all findings, likely-real only, and confirmed-real (manually validated subset).
  - **Attribution accuracy:** Manually verify a random sample (n=200) of AI-attributed commits to confirm the attribution signal is accurate. Report the false-attribution rate and adjust estimates accordingly.
  - **Confounders:** Control for file type (config files vs. source code vs. tests), diff size, repository star count (proxy for quality), and commit recency. Use logistic regression with these covariates to isolate the AI vs. human effect.
  - **Statistical approach:** Primary metric: secrets per 1,000 lines of added code, AI vs. human. Two-sample proportion test with Bonferroni correction for multiple secret types. Pre-register the analysis plan to avoid p-hacking accusations.

- **Timeline:** 5-6 weeks. Week 1-2: commit collection, attribution classification, and matched-pair construction. Week 3: automated scanning of 200,000 diffs. Week 4: placeholder classification, manual validation of subsamples. Week 5: statistical analysis and regression modeling. Week 6: write-up.

**Expected Findings:** The honest answer is that we do not know which direction this will go, and that uncertainty is what makes it publishable. Plausible outcomes:
- **AI leaks more:** AI tools may generate code with example credentials from training data, or users may paste secrets into prompts that get echoed back into code. If so, we expect the difference to be concentrated in specific patterns (generic API keys, bearer tokens, database connection strings) rather than platform-specific tokens (GitHub PATs, AWS keys).
- **AI leaks less:** AI tools may be trained to avoid generating real-looking secrets, and human developers are the ones who actually have credentials to accidentally commit. If so, the result is still newsworthy as a vindication.
- **No significant difference:** Also publishable and useful -- it means AI tools are neither better nor worse than humans at secret hygiene, which has policy implications.

Regardless of direction, the per-secret-type breakdown will reveal which patterns are most associated with AI-generated code, which is actionable for both AI tool developers and security teams.

**Publication Strategy:**
- Primary: Academic preprint on arXiv (cs.CR or cs.SE), targeting submission to IEEE S&P, USENIX Security, or CCS
- Secondary: Accessible blog post summarizing key findings with visualizations
- Amplification: Coordinate with AI tool vendors (GitHub Copilot team, Anthropic, Cursor) for comment/response before publication. Pitch exclusive to Wired or Ars Technica for mainstream coverage.
- Data release: Publish the full analysis pipeline (Rafter commands, matching algorithm, regression specification) and aggregate dataset. Individual commits identified only by anonymized hash.

**Ethical Considerations:**
- **Public data only.** All commits are from public repositories. No private repository access.
- **No secret exploitation.** Discovered secrets are never tested for validity.
- **Responsible disclosure to AI vendors:** Share pre-publication findings with GitHub (Copilot), Anthropic (Claude), Cursor, and OpenAI (Codex) so they can investigate whether their tools contribute to the problem and implement mitigations.
- **Avoid blaming individual developers.** Results are reported as aggregate rates, never tied to specific users or repositories.
- **IRB consideration:** While public GitHub data may not require IRB review, consult with institutional review if pursuing academic publication. The study involves observable public behavior with no intervention.

## 3. The Phantom Toolbox: Mapping Trust Boundaries in MCP Server Ecosystems

### Project Title
**The Phantom Toolbox: Auditing Supply Chain and Trust Boundary Violations in MCP Server Registries**

**Research Question:** When AI agents connect to third-party MCP servers, what is the actual security posture of the tool implementations they execute -- and how many MCP servers in public registries contain hidden capabilities, credential harvesting code, or supply chain risks that violate the trust assumptions of the agents consuming them?

**Why This Matters:** The Model Context Protocol (MCP) is rapidly becoming the standard interface between AI agents and external tools. Public MCP server registries (mcp.run, Smithery, glama.ai, individual GitHub repos) now list thousands of servers that any agent can connect to. But MCP has no built-in sandboxing, no code signing, and no mandatory security review. An MCP server is arbitrary code that runs with the permissions of the user's machine, often invoked automatically by an AI agent without explicit user approval of each tool call. This is structurally identical to the early npm/PyPI supply chain problem, but worse: the consumer is an AI agent that cannot reason about whether a tool implementation is safe. No systematic audit of MCP server security has been published. This study would be the first.

**Methodology:**

- **Data collection:** Enumerate MCP servers from three source categories:
  - **Public registries:** Crawl mcp.run, Smithery.ai, and glama.ai for all listed MCP server packages. Expected: 2,000-5,000 servers.
  - **GitHub discovery:** Search GitHub for repositories containing MCP server implementations (identifiers: `McpServer`, `createServer` from `@modelcontextprotocol/sdk`, `mcp.server`, `FastMCP`, `mcp-server` in package.json/pyproject.toml). Expected: 3,000-8,000 repositories.
  - **Package registries:** Search npm for packages with `mcp-server` or `mcp` keywords, PyPI for packages with `mcp` classifier or keyword. Expected: 500-2,000 packages.

  For each server, clone the source repository and collect: source code, declared tool definitions (names, descriptions, input schemas), dependency manifests (package.json, requirements.txt, pyproject.toml), and any configuration templates.

- **Analysis tools (multi-layer audit):**

  **Layer 1 -- Secret scanning with Rafter:**
  - `rafter scan local --format json` on every MCP server repository. This catches hardcoded credentials in server implementations (database passwords, API keys used by the server itself, leaked tokens in test fixtures).
  - `rafter agent scan` on any configuration templates or example configs shipped with the server. These often contain real credentials left by the developer.

  **Layer 2 -- Capability analysis (custom tooling, built on Rafter's pattern engine):**
  - Parse each server's tool definitions to extract declared capabilities. Map the tool taxonomy: file system access, network requests, shell execution, credential management, database access.
  - Compare declared capabilities (tool descriptions) against actual code behavior. Flag "capability mismatches" where a tool described as "read a file" also makes network requests, or a tool described as "search the web" also writes to the filesystem.
  - Detect "hidden tools" -- tools registered with the MCP server but not advertised in the README or registry listing.

  **Layer 3 -- Dependency supply chain analysis:**
  - For each MCP server, resolve the full dependency tree (npm: `npm ls --all --json`, Python: `pip-audit` or `safety`).
  - Cross-reference against known vulnerability databases (OSV, GitHub Advisory Database).
  - Measure dependency freshness (days since last update) and maintainer count.
  - Flag servers that vendor or bundle dependencies (hiding the supply chain from standard audit tools).

  **Layer 4 -- Behavioral red flags (static analysis):**
  - Scan server source code for patterns indicating credential harvesting: reading environment variables beyond what tool functionality requires, exfiltrating data to external endpoints, accessing credential stores (`~/.aws/credentials`, `~/.ssh/`, browser cookie stores, keychain APIs).
  - Scan for obfuscated code (base64 encoded strings, eval/exec calls, dynamic imports from URLs).
  - Scan for persistence mechanisms (cron jobs, startup scripts, file watchers that survive the MCP session).

- **Controls:**
  - **False positive reduction for capability mismatches:** A "mismatch" is only flagged if the undeclared capability involves a sensitive operation (network, filesystem write, shell exec, credential access). Read-only operations and logging are excluded.
  - **Baseline for dependency risk:** Compare MCP server dependency profiles against a matched sample of non-MCP npm/PyPI packages of similar size and age. This establishes whether MCP servers are unusually dependency-heavy or dependency-stale.
  - **Manual triage:** All Layer 4 behavioral red flags are manually reviewed by a security researcher before being classified as "suspicious" or "benign." Target manual review of 100% of red-flagged servers (expected: 50-200 servers).
  - **Reproducibility:** Publish the full scanning pipeline as a Rafter plugin/recipe so others can re-run the audit as registries grow.

- **Timeline:** 8-10 weeks. Weeks 1-2: registry crawling, repository collection, deduplication. Weeks 3-4: Layer 1 (Rafter secret scanning) and Layer 3 (dependency analysis) -- these are fully automated. Weeks 5-6: Layer 2 (capability analysis) -- requires building the tool-definition parser and capability-mismatch detector. Weeks 7-8: Layer 4 (behavioral red flag scanning) and manual triage. Weeks 9-10: statistical analysis, visualization, write-up.

**Expected Findings:**
- **Secrets in MCP servers:** Based on rates seen in npm/PyPI packages generally, we expect 3-8% of MCP server repositories to contain at least one hardcoded secret, with higher rates in early-stage single-developer projects.
- **Capability mismatches:** This is the most uncertain and potentially most impactful finding. We expect to find servers where the declared tool description understates actual capabilities -- a "weather lookup" tool that also reads local files, a "code formatter" that makes outbound HTTP requests. Even a small number of these (dozens) would be significant because it demonstrates that agents cannot trust tool descriptions.
- **Dependency risk:** We expect MCP servers to have shallower dependency trees than typical web applications but higher rates of unmaintained dependencies, because many MCP servers are weekend projects that were published and abandoned.
- **Behavioral red flags:** We expect to find a small number (5-20) of servers with genuinely suspicious behavior patterns (credential access, data exfiltration endpoints, obfuscated code). Whether these are malicious or merely poorly written is a judgment call that the manual review will resolve.

**Publication Strategy:**
- Primary: Detailed technical report (15-20 pages) suitable for both blog publication and conference submission. Target USENIX Security 2027 or IEEE S&P 2027 for academic venue.
- Secondary: A "State of MCP Security" summary report with key statistics and recommendations, targeting CISOs and security teams evaluating MCP adoption.
- Tool release: Open-source the MCP server auditing pipeline as a Rafter extension, so the community can continuously monitor the ecosystem.
- Amplification: Coordinate with the MCP specification authors (Anthropic) and registry operators (mcp.run, Smithery) before publication. Present findings at DEF CON AI Village or Black Hat Arsenal if accepted.
- Media: Pitch to security-focused outlets (The Record, Krebs on Security, Risky Business podcast) with the angle "the npm supply chain crisis is happening again, and this time the consumer is an AI that can't say no."

**Ethical Considerations:**
- **Responsible disclosure:** Any MCP server found to contain likely-malicious code (credential harvesting, data exfiltration) will be reported to the registry operator and the developer via private disclosure BEFORE publication. Standard 90-day disclosure timeline.
- **No exploitation:** We do not execute MCP servers or test discovered capabilities. All analysis is static (source code review).
- **Registry notification:** Notify mcp.run, Smithery, and glama.ai of aggregate findings before publication so they can implement security review processes.
- **Naming policy:** Servers with confirmed malicious behavior are named (this is standard practice for supply chain security research). Servers with mere poor hygiene (stale dependencies, hardcoded test credentials) are reported in aggregate only.
- **Scope limitation:** We audit publicly available MCP servers only. No social engineering, no credential testing, no active exploitation.
